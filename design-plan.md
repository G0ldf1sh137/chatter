Users register (username/password or Google) after verifying their email, post in Markdown, comment with unlimited nesting, upvote/downvote posts and comments, follow each other, maintain a profile with an avatar, bio, and karma, and play simple games â€” both single-player and turn-based multiplayer â€” against each other. ## 1. Goals and scope

Posts and comments are authored in Markdown and render as formatted HTML when published (see Â§5.1). Pages are server-rendered with Django templates â€” no REST API, no JavaScript framework, and no Node.js anywhere in the stack (styling uses Tailwind CSS via its standalone CLI binary, not npm â€” see Â§5.3); the deliberate exceptions are 2048/Snake/Doodle Jump/Flappy Bird/Tetris, where a real client-side game loop is unavoidable, a handful of small, focused vanilla-JS enhancements (infinite scroll, match polling, the your-turn/unread-message badges â€” Â§5.6, Â§5.7, Â§5.8) that don't rise to needing a framework, and the `simulations` app's p5.js sketches, loaded from a CDN rather than vendored (Â§5.9). Still out of scope: group messaging (direct messages are strictly one-to-one â€” Â§5.8), image embedding inside Markdown post/comment bodies (avatars are the only image upload), full-text search, moderation tools, and genuinely real-time multiplayer (games are turn-based/asynchronous by design, kept fresh with lightweight polling rather than a push channel â€” see Â§5.4, Â§5.7). ## 2. Tech stack

Python 3.12, Django 5.x, PostgreSQL 16 (official Docker image), psycopg 3 as the database driver, Gunicorn as the app server inside the container, and Django's template engine styled with Tailwind CSS (standalone CLI, no Node/npm â€” see Â§5.3). Markdown rendering uses `markdown-it-py` (CommonMark-compliant) with `nh3` for HTML sanitization and `Pygments` for server-side syntax highlighting of fenced code blocks (Â§5.1). `django-allauth` provides Google OAuth alongside the built-in username/password auth. Email verification is hand-rolled (a signed, time-limited token via `django.core.signing`, not allauth's own email-confirmation flow â€” see Â§5.1a) since it only needs to gate the site's own registration form, not allauth's. `Pillow` backs the avatar `ImageField`. Dependencies are managed with `uv`: declared in `pyproject.toml`, locked in `uv.lock` (committed), so every environment â€” dev, Docker build, CI â€” installs the exact same versions with `uv sync`. Configuration (secret key, debug flag, database URL, Google OAuth credentials, SMTP settings) comes from environment variables so the same image works in dev and in build tests.

Three deployment-oriented dependencies were added alongside the Render/Neon work (Â§6a) without touching the local dev experience at all: `dj-database-url` parses Neon's single `DATABASE_URL` connection string into Django's `DATABASES` shape, `whitenoise` serves `collectstatic`'s output directly from the Gunicorn process (no separate static host), and `django-storages` provides an optional S3-compatible avatar backend for hosts with ephemeral disk. All three are no-ops in the Docker Compose/SQLite dev setup, which never sets `DATABASE_URL`/`DEBUG=0`/`AWS_STORAGE_BUCKET_NAME`.

## 4. Data model

Every model this app owns (`Profile`, `Follow`, `Post`, `Comment`, `PostVote`, `CommentVote`, `Match`, `SinglePlayerResult`) uses a `UUIDField(default=uuid.uuid4, editable=False)` primary key instead of Django's default auto-incrementing integer, so IDs are unguessable and non-enumerable in URLs (`/posts/<uuid>/`, not `/posts/1/`). `auth.User` itself was deliberately left out of this and still uses an integer PK â€” switching it would mean replacing Django's built-in user model with a custom one, which touches sessions, admin, and allauth's `SocialAccount` linkage and is explicitly the kind of decision Django warns against making mid-project; the tradeoff (User is the one model in this app whose numeric ID is exposed nowhere in a URL anyway) wasn't worth that risk. Converting existing integer PKs to UUIDs turned out to be impossible to express as a normal migration on Postgres â€” there's no valid cast from `bigint` to `uuid`, so `ALTER COLUMN id TYPE uuid` fails outright even against an empty table â€” so the `posts`/`accounts`/`games` migration histories were squashed to a single fresh `0001_initial` per app instead, which requires resetting any existing database (including in production) when deploying this change.

**User** â€” Django's built-in `auth.User` (username, password hash, email, joined date). Email is required at registration (`RegistrationForm` extends `UserCreationForm`); Google-created accounts get a real email from Google automatically.

**Profile** â€” one-to-one with User. Fields: `bio` (text, optional), `avatar` (`ImageField`, optional, capped at `MAX_AVATAR_UPLOAD_SIZE` â€” 2MB â€” validated in `ProfileForm.clean_avatar`), `email_verified` (bool, default `False`), `timezone` (a `CharField` restricted to `choices=` built from `zoneinfo.available_timezones()`, default `"UTC"` â€” see Â§5.10), `created_at`. Created automatically via a `post_save` signal on User (`accounts/signals.py`) â€” this covers both registration paths, since Google sign-in also creates a standard `User` row through the same manager. Accounts that existed before `email_verified` was added were grandfathered in as verified by a one-off data migration, so the gate only applies to registrations from that point on.

**Follow** â€” `follower` (FK â†’ User, `related_name="following"`), `followed` (FK â†’ User, `related_name="followers"`), `created_at`. A `UniqueConstraint` on `(follower, followed)` prevents duplicate follows; a `CheckConstraint` prevents self-follows at the database level, not just in the view.

**Post** â€” `author` (FK â†’ User, `on_delete=CASCADE`), `body` (text, max ~5,000 chars), `edited` (bool, default `False` â€” see Â§5.5), `created_at`, `updated_at`. Default ordering is `-created_at`, but the feed itself orders by score (see Â§5).

**Comment** â€” `author` (FK â†’ User), `post` (FK â†’ Post, always set, even for nested replies), `parent` (FK â†’ self, nullable; null means top-level), `body` (text), `edited` (bool, default `False` â€” see Â§5.5), `created_at`.

**PostVote** / **CommentVote** â€” both subclass an abstract `Vote` base (`user`, `value` â€” `+1`/`-1` via `Vote.UP`/`Vote.DOWN`, `created_at`) rather than a single generic-relation model, since there are only ever two votable types and duplicating a three-field model twice is simpler than `ContentType`/`GenericForeignKey` machinery. Each has a `UniqueConstraint` on `(user, post)` / `(user, comment)` â€” one vote per user per item, flipped or removed by voting again rather than accumulating rows.

Threading strategy (unchanged from v1): fetch all comments for a post in one query, build the tree in Python (`build_comment_tree`, a dict of `parent_id â†’ children`). O(n), avoids recursive SQL.

**Conversation** â€” `user1`/`user2` (FK â†’ User), `created_at`. Always stored with `user1_id < user2_id` (canonicalized in `get_or_create_conversation()`, not left to whichever order the two users happen to be passed in), so a `UniqueConstraint` on `(user1, user2)` can guarantee exactly one conversation per pair of users regardless of who messaged whom first - an unordered pair can't be uniqued directly. A `CheckConstraint` blocks messaging yourself, same pattern as `Match`'s self-challenge guard.

**Message** â€” `conversation` (FK â†’ Conversation), `sender` (FK â†’ User), `body` (text), `read` (bool, default `False`), `created_at`. `read` means "has the *other* participant read this" - a conversation only ever has two people, so there's no need for a per-recipient read-receipt table the way a group chat would need. See Â§5.8.

**Match** â€” one generic model for all five multiplayer games rather than a table per game, since Tic-Tac-Toe, Rock-Paper-Scissors, Connect Four, Checkers, and Othello all share an identical shape (2 players, one state blob, one winner) â€” unlike Post/Comment, which are genuinely different parent objects and justify `Vote`'s abstract-base-plus-subclass split above. Fields: `game` (`"ttt"`/`"rps"`/`"connect4"`/`"checkers"`/`"othello"`), `player1`/`player2` (FK â†’ User), `state` (`JSONField`, game-specific shape â€” a 9-cell board array for Tic-Tac-Toe, a `{"choices": {...}}` map keyed by user ID for Rock-Paper-Scissors' simultaneous picks, a 6Ă—7 nested array for Connect Four, an 8Ă—8 nested array of `"r"`/`"b"`/`"R"`/`"B"` piece codes for Checkers, an 8Ă—8 nested array of `"B"`/`"W"` piece codes for Othello), `status` (`"active"`/`"finished"`), `turn` (FK â†’ User, nullable â€” null for Rock-Paper-Scissors, which has no "whose turn" concept), `winner` (FK â†’ User, nullable; null on a finished match means a draw), timestamps. A `CheckConstraint` prevents challenging yourself. No constraint blocks multiple concurrent matches between the same two players (an immediate rematch is fine).

**SinglePlayerResult** â€” one row per completed single-player game. Fields: `player` (FK â†’ User), `game` (`"hangman"`/`"2048"`/`"snake"`/`"doodle"`/`"wordle"`), `won` (bool), `score` (int), `created_at`. In-progress single-player state doesn't get a DB row: Hangman and Wordle use `request.session` (there's already a precedent for session-scratch-state in `accounts/views.py`'s `pending_verification_email`), and 2048/Snake/Doodle Jump are entirely client-side; a `SinglePlayerResult` is only written once a game actually ends. Snake and Doodle Jump have no natural win condition (both are endless score-chasers), so their `Finish` views always record `won=False` â€” score is the only meaningful signal, same as 2048's leaderboard ranking by high score rather than win count. Wordle's `score` is guesses-remaining-after-a-win (6 for a first-guess win, 1 for a last-guess win, 0 for a loss) rather than a flat win/loss flag like Hangman â€” framing it as "higher is better" lets it reuse the same descending-high-score leaderboard shape without inventing an ascending "fewest guesses" ranking.

Deletion policy (unchanged): deleting a user cascades to their posts, comments, votes, and follows.

### 5.1 Markdown rendering

Unchanged from v1: post and comment bodies are stored as raw Markdown, never rendered HTML. `markdown-it-py` (CommonMark plus fenced code blocks, tables, strikethrough, and linkify for bare URLs) converts to HTML, then `nh3` strips everything outside an allowlist of tags/attributes â€” this is what makes user-authored Markdown safe against `<script>` tags, `javascript:` hrefs, and event-handler attributes. Links get `rel="nofollow noopener"`. Rendered via a single `render_markdown` template filter shared by posts, comments, and direct messages (Â§5.8). Still no image embedding via Markdown.

Fenced code blocks with a language tag (` ```python `, not a bare ` ``` `) get server-side syntax highlighting via Pygments (`posts/markdown.py`'s `_highlight_code`, passed to `MarkdownIt` as its `highlight` hook) rather than a client-side JS highlighter, keeping with the "no JS framework" approach used everywhere else. A bare fence with no language falls through to plain escaped `<pre><code>` rather than guessing a lexer from content alone, which would risk mislabeling plain text/output as some unrelated language. `nh3`'s allowlist grew `span` (Pygments wraps each token in one) and a `class` attribute on `pre`/`code`/`span`, since those classes are what carry the actual highlight colors - stripping them would silently discard the highlighting while leaving the code text intact. The token-color CSS itself (`static/css/input.css`) uses Pygments' "one-dark" style unconditionally, not swapped per light/dark page theme: Tailwind Typography's own `pre` background (`--tw-prose-pre-bg`/`--tw-prose-invert-pre-bg`) turns out to already be a fixed dark shade in *both* prose and `prose-invert` - code blocks are conventionally dark regardless of the surrounding page - so a light Pygments palette would have rendered dark-on-dark and been discovered as a real contrast bug during manual review, not by construction.

### 5.1a Email verification

Registering with username/password no longer logs the user in immediately: `RegisterView` emails a signed, expiring link (`django.core.signing`, 3-day max age via `EMAIL_VERIFICATION_MAX_AGE`) and redirects to a "check your email" page. Logging in is blocked until the link is visited â€” enforced in `EmailVerifiedAuthenticationForm.confirm_login_allowed` (`accounts/forms.py`), the officially-supported Django hook for exactly this kind of gate, rather than fighting with allauth's backend internals. `EMAIL_BACKEND` defaults to Django's console backend in dev (no `EMAIL_HOST` set), so links show up in the server log without needing real SMTP.

Google sign-in is exempt, since Google already verified the email: `accounts/signals.py` hooks allauth's `user_signed_up` signal (which only fires with a `sociallogin` kwarg for social accounts, never for the plain `RegisterView`) to mark those profiles verified immediately. Separately, `accounts/adapter.py`'s `SocialAccountAdapter.pre_social_login` looks up an existing user by the Google-provided email *before* allauth decides whether to show its own signup form, and connects the login to that existing account if one matches â€” treating a Google-verified email as proof of ownership rather than showing allauth's default (unstyled) conflict-resolution form.

### 5.1b Account settings

`/settings/profile/` (`ProfileEditView`) edits `Profile` (bio/avatar) and `User` (username/first/last name) together, one submit for both â€” `UserProfileForm` binds to a *fresh copy* of the `User` row fetched by primary key, not `request.user` directly, because Django's `ModelForm` writes submitted values onto its bound instance during validation *before* a uniqueness check can fail it. Binding `request.user` itself meant a rejected duplicate-username submission would leak the not-yet-saved, rejected username onto the same object `_header.html`'s nav link reads from, pointing "Log out"'s neighbor at a different real user's profile for the rest of that one response â€” a real bug caught by testing the duplicate-username path specifically, fixed by giving the form its own copy to mutate.

`/settings/password/` (`PasswordChangeView`) picks between Django's `PasswordChangeForm` (requires the current password) and `SetPasswordForm` (no current-password check) based on `request.user.has_usable_password()` â€” Google-only signups get an unusable password from allauth, so they'd never be able to satisfy `PasswordChangeForm`'s check otherwise. `update_session_auth_hash` keeps the session logged in afterward, same as Django's own built-in password-change view does.

### 5.2 Voting and karma

Clicking a vote arrow toggles it: clicking the same direction again removes the vote, clicking the opposite direction flips it (`toggle_vote` in `posts/views.py`). A `post_save` signal (`posts/signals.py`, guarded on `created=True`) automatically gives every new post and comment a self-upvote from its author â€” this fires regardless of how the row is created (view, admin, shell), not just through `PostCreateView`/`CommentCreateView`.

Score and "did the current user vote, and which way" are computed via `annotate_votes()`, shared across the feed, post detail, and profile post/comment lists: `Coalesce(Sum("votes__value"), 0)` for score, a correlated `Subquery` against the vote table for the current user's vote direction. One gotcha worth documenting: aggregation via `.annotate()` silently drops a queryset's ordering (explicit or the model's default `Meta.ordering`) from the generated SQL, so `annotate_votes()` re-captures and reapplies whatever ordering was present beforehand.

Karma on the profile page is the sum of vote values across all of a user's posts and comments (two aggregate queries, `PostVote`/`CommentVote` filtered by `post__author`/`comment__author`).

The feed is no longer sorted by raw post score. `posts/ranking.py`'s `rank_posts()` blends four signals with tunable weights (`POST_KARMA_WEIGHT=1.0`, `COMMENT_KARMA_WEIGHT=0.5`, `POSTER_KARMA_WEIGHT=0.1`): the post's own karma, the karma earned by its comment thread, and the poster's overall karma (their total across all posts and comments, profile-page karma reused as-is) â€” then decays the weighted total by the post's age using a Hacker News-style `weighted_karma / (age_hours + 2) ** GRAVITY` formula (`GRAVITY=1.8`), so a post's rank fades over time even if its karma never changes and a recent post can outrank an older, higher-karma one once it's aged enough.

This runs in Python after a single fetch rather than as one big `annotate()`, deliberately: summing two different reverse relations (`PostVote` and `CommentVote`, reached via two different join paths from `Post`) in the same `annotate()` call causes a join fan-out that silently inflates both sums â€” a sharper version of the ordering gotcha above. `rank_posts()` sidesteps it with separate grouped-aggregate queries (comment karma per post, post/comment karma per author) merged into Python, then sorts the fetched page's posts by the computed score. The trade-off: ranking is recomputed over every matching post on each feed load rather than pre-sorted at the database level â€” proportionate at this app's scale, but worth revisiting (e.g. a cached/precomputed rank) if post volume grew substantially (see Â§10).

The default ranking isn't the only option: a `?sort=` query param (`FeedView.get_sort()`, whitelisted against `{"default", "top", "new"}`, falling back to `"default"` for anything else) switches between it, `top` (`order_by("-score", "-created_at")` â€” the plain popularity ordering the feed used before ranking existed), and `new` (`order_by("-created_at")`, ignoring votes entirely). `feed.html` exposes this as a second pill-button row next to the All/Following tabs, and both the feed-type links and pagination links carry the current `sort` value forward so switching feeds or pages doesn't silently reset it back to `default`.

### 5.3 Styling

Tailwind CSS, via the [standalone CLI binary](https://tailwindcss.com/blog/standalone-cli) rather than a Node/npm toolchain â€” the Dockerfile downloads a pinned, architecture-aware binary the same way it already vendors the `uv` binary, so the stack stays Python-only. Source is `static/css/input.css`; compiled output is `static/css/tailwind.css` (gitignored, generated). Markdown-rendered post/comment bodies use the bundled `@tailwindcss/typography` plugin (the `prose` class) instead of hand-styling arbitrary HTML â€” the standalone CLI ships the official first-party plugins without needing npm. The reply/edit-toggle "checkbox hack" uses Tailwind's named `peer`/`peer-checked` variants (`peer/reply`, `peer/edit` â€” see Â§5.5), so a comment showing both toggles at once doesn't have one checkbox's state leak into the other's form.

The color system ("Cobalt Current" â€” cobalt blue brand, amber upvotes, rose downvotes, emerald karma) is a set of semantic tokens (`--bg`, `--surface`, `--fg`, `--accent`, ...) defined once on `:root` and remapped under `.dark`, then exposed to Tailwind via `@theme { --color-bg: var(--bg); ... }` â€” templates use `bg-surface`, `text-accent`, etc. throughout rather than raw Tailwind colors or `dark:` variants, so a full palette swap (as happened once already, from an earlier violet/indigo palette) only touches the two variable blocks in `input.css`, never the ~20 templates that reference them. Dark mode is class-based (`@custom-variant dark (&:where(.dark, .dark *));`) rather than Tailwind's `prefers-color-scheme` default, since the header's sun/moon toggle button needs to override the OS preference and persist the choice â€” a blocking inline script in `base.html`'s `<head>` (before the stylesheet paints) applies `.dark` from `localStorage`, falling back to `prefers-color-scheme` only on a first visit, which avoids a flash of the wrong theme that a deferred/post-paint script wouldn't.

The comment form is inline per-comment rather than a single shared form at the bottom of the page: each comment has its own collapsible reply form, expanded via a hidden checkbox and a `<label>`, with no JavaScript â€” matching the "no JS framework" goal while avoiding the earlier design's jump-to-bottom-of-page UX.

Voting redirects back to a full page reload with posts re-sorted by score, which would otherwise just snap to the new order. `input.css` opts every navigation into the browser's native cross-document View Transitions API (`@view-transition { navigation: auto; }` â€” Chrome/Edge 126+, a harmless no-op elsewhere), and `feed.html` gives each post card a stable `view-transition-name` (`post-<id>`), so the browser matches a card across the old and new page and animates it sliding to its new rank instead of jumping. No custom JS.

### 5.4 Games

The `games` app adds Tic-Tac-Toe, Rock-Paper-Scissors, Connect Four, Checkers, and Othello (turn-based multiplayer, challenged from a profile page like the Follow relationship â€” no open lobby, no accept/decline handshake), Word Guess/Hangman and Wordle (single-player, session-based), and 2048, Snake, and a Doodle Jump clone (single-player, client-side).

Multiplayer is explicitly **turn-based/asynchronous, not real-time** â€” a deliberate choice to avoid adding Channels/an ASGI server/a Redis channel layer for a feature that doesn't need them. A move is a normal POST + redirect; the opponent sees it next time they load the page. `games/views.py`'s `GamesHubView` groups a player's active matches into "your turn" vs "waiting on opponent" so discovering a pending move doesn't require a notification system. Concurrent move submissions (mainly a concern for Rock-Paper-Scissors, where both players can legitimately move at once) are handled with `select_for_update()` inside `transaction.atomic()`, re-checking `status`/`turn` after acquiring the lock â€” not a queue or external lock, which would be over-engineering at this scale.

Game rules live in `games/logic/` as pure Python functions with no Django imports (mirroring `posts/markdown.py` as the precedent for keeping rules separate from the view/model layer), directly unit-tested without touching the database. Connect Four's win detection scans all four line directions from every occupied cell. Checkers uses a deliberately simplified ruleset (diagonal moves only, captures optional rather than forced, single jump per turn, kings move any diagonal direction) rather than standard American forced-capture rules â€” a non-king piece can still capture backward even though it can't simple-move backward, the standard convention, documented explicitly in the module since the simplified spec left it open; there's no draw concept since the lack of forced captures means no mutual-stalemate deadlock, only a one-sided "no legal moves" loss. Checkers' move UX is inherently two clicks (select a piece, then a destination), handled by a small vanilla-JS click helper (`checkers.js`) that fills two hidden form fields and submits â€” simpler than tracking a "selected piece" in server-side session state, and consistent with the project's "interaction logic in JS, server stays a dumb state-transition validator" split.

Othello is the one game where turn order isn't strict alternation: a move must flip at least one opponent piece (`games/logic/othello.py`'s `_flips_for_move` scans all 8 directions, collecting a contiguous opponent run and only committing the flip if it terminates on the mover's own color before the board edge or an empty cell), and a player with no legal move must pass back to their opponent rather than the game ending â€” it only ends once *neither* player has a legal move, with the winner decided by piece count. `next_turn_state(state, mover, opponent)` centralizes this as a single three-way decision (`"continue"`/`"pass"`/`"game_over"`), mirroring how `checkers.check_winner(state, next_player)` already centralizes a similar "is the next player stuck" check, so the view only has to translate the result into a `Match.turn`/`status`/`winner` update. The one UX wrinkle: a player can legitimately see "Your turn" twice in a row if their opponent just passed, so the match template derives a `just_passed` flag (opponent currently has zero legal moves) to show "so-and-so had no legal move and passed" instead of a bare turn banner.

2048, Snake, Doodle Jump, Flappy Bird, and Tetris are the exceptions to "no client-side game logic": their rules live entirely in `games/static/games/js/`, vanilla JS game loops, since real-time gameplay can't be a server round-trip per move. 2048 implements all four slide directions with a single "slide left" routine plus a rotate-and-rotate-back trick; each power-of-two tile value gets its own shade by cycling through the theme's four semantic accent colors (Â§5.3) â€” blue, amber, rose, green, back to solid blue at 4096 â€” rather than the two flat color bands it originally shipped with, which left every tile below 64 rendered as `bg-surface-hover`, identical to the board's own background and effectively invisible. Snake uses a DOM grid re-rendered each tick (`setTimeout` chaining so the tick rate can speed up as the snake grows). Doodle Jump is the app's first `<canvas>`-based game, using `requestAnimationFrame` for continuous gravity/jump physics with static scrolling platforms (no moving/breakable platforms or enemies â€” kept to a simple MVP). Flappy Bird reuses that same canvas/RAF/gravity architecture wholesale â€” scrolling pipes with randomized gap positions stand in for Doodle Jump's scrolling platforms, and "flap" is just a fixed upward velocity impulse on click/tap/space instead of left/right movement - proof the canvas-game pattern Doodle Jump established generalizes to a second game with almost no new architecture, only new obstacle/collision shapes. Tetris (`tetris.js`) goes back to Snake's discrete-grid/tick-chained shape instead of continuous canvas physics, since Tetris moves in whole cells on a fixed timer rather than sub-pixel motion, but it's the first client-side game needing genuinely new logic rather than a reskin: a two-layer board (a locked grid of settled blocks plus a currently-falling piece tracked separately and overlaid at render time), the seven tetromino shapes and their four rotation states hardcoded as literal coordinate data (universal, exactly-documented reference data, so - unlike Nine Men's Morris' generated adjacency table - there's no algorithm worth writing just to reproduce it), one `collides()` check reused for every action (move, rotate, and the automatic fall-tick all just ask "would this candidate position overlap the board edge or a locked cell"), and line-clearing (scan every row for "completely filled," splice it out, unshift an empty row at the top, score via the classic simultaneous-clear bonus table so a 4-line "Tetris" is worth much more than four separate singles). Wall-kicks (nudging a rotation that would otherwise collide) are deliberately cut - a rotation near a wall or another piece just fails outright - the same simplification posture as Backgammon's dropped doubling cube. All five client-side games POST just a final score (2048 also sends the highest tile reached) to a `FinishView` at game over; the server does bounds-checking (sane ranges, and for 2048 a power-of-two tile check consistent with the score) rather than full move-replay validation, proportionate to a casual game's stakes.

Wordle (`games/logic/wordle.py`) follows Hangman's session-based pattern instead â€” no client-side JS, since a whole-word guess is naturally a single form submission. `apply_guess` does a two-pass comparison against the target: an exact-match pass marks correct letters and removes them from a per-letter count pool built from the target, then a second pass marks remaining guess letters "present" only while the pool still has that letter available, else "absent" â€” the classic fix for duplicate-letter guesses (e.g. guessing "apple" against target "grape" correctly marks only the first of the two guessed P's "present", since "grape" has just one). Unlike Hangman's always-valid `apply_guess` (invalidity is filtered before the call), Wordle's raises `InvalidMove` for a guess that isn't a 5-letter word in its own `WORD_LIST` â€” the same list doubles as the valid-guess dictionary, proportionate scope rather than maintaining a separate larger dictionary.

Nim (`games/logic/nim.py`) needed no new mechanic at all â€” three piles, strict alternating turns, last player to take a stick wins â€” which is exactly why it was picked as a quick addition: `stats.match_win_leaders`/`match_record` and the your-turn/polling infrastructure (Â§5.7) all worked unmodified, and the only view-layer work was precomputing each pile's clickable "take N sticks" buttons as ready-to-iterate data (`piles[].sticks[].remove_count`) rather than asking the template to do the arithmetic, the same fix already established for Othello's `board_cells`.

Mastermind (`games/logic/mastermind.py`) reuses Wordle's "set aside exact matches, then count what's left" duplicate-handling shape (Â§5.4 above), just over aggregate color counts instead of per-position letter status: black pegs are exact position+color matches, white pegs are correct colors in the wrong position, counted from what's left after black pegs are set aside so a repeated color in the guess can't earn more pegs than the secret actually has of that color. Session-based like Wordle, and its score (guesses-remaining-after-a-win) reuses the same descending high-score leaderboard shape for the same reason Wordle's does.

Battleship (`games/logic/battleship.py`) is the app's first game where `Match.state` can't be shown to both players unmodified â€” every prior game's state is symmetric (both players are allowed to see the whole board), but a player's ship layout must stay hidden from their opponent. Placement is its own sub-phase (`state["phase"] == "placement"`) before battle starts: each player independently places a simplified 4-ship fleet (lengths 4/3/2/2 on an 8x8 board, smaller than the classic 5-ship/10x10 set to keep placement to 4 plain form submissions per player and the grid renderable without JS) via `apply_placement`, and `match.turn` stays `None` until `both_players_placed` is true â€” the same "no turn concept yet" shape Rock-Paper-Scissors' simultaneous-choice phase already uses, right down to `stats.is_users_turn` getting a matching special case (placement-phase Battleship means "you still have ships left to place," not "it's your move"). Once battle starts, turns alternate strictly like Nim/Tic-Tac-Toe. The redaction itself is `battleship.viewer_state(state, viewer_id, opponent_id)`, a pure function returning only the viewer's own ship layout plus which of the viewer's shots against the opponent were hits or misses â€” the opponent's un-shot ship cells never appear in its output at all, so `BattleshipMatchView` builds its template context exclusively from that redacted structure rather than touching `match.state` directly. This is the one generalizable takeaway for any future hidden-information game: a `viewer_state(state, viewer_id)` function in the logic module, with the view required to pass its result to the template instead of the raw state, the same way every other game's view is required to precompute template-unfriendly arithmetic in Python rather than asking the template to do it.

Stratego (`games/logic/stratego.py`) extends that same `viewer_state` contract with a different slice of "hidden": Battleship hides a cell's contents entirely (position *and* identity) until it's fired upon, but Stratego's piece *positions* are public from turn one â€” both players see the whole board â€” while only a piece's *rank* stays secret until it fights, and once revealed by combat that reveal is permanent (tracked as a `revealed` flag per piece, not per event, since a piece's `id` has to stay stable across moves for that to mean anything). `viewer_state(state, viewer_id, opponent_id)` returns every opponent piece's row/col unconditionally but its rank only if `revealed` is true â€” the regression test that matters here is the mirror image of Battleship's: confirming `row`/`col` are always present for an opponent piece while an unrevealed one's `rank` never is. A simplified ~10-piece fleet (ranks 1â€“6 plus a Spy, a Miner, two Bombs, and a Flag) replaces the classic 40-piece/10x10 set, and movement is one square orthogonally only (no Scout long-range move) - the same proportionality tradeoff as Checkers' optional-capture ruleset - but the rules that make it recognizably Stratego stay intact: rank beats rank, the Spy-beats-Marshal exception only when the Spy attacks, a Bomb defeats anything but a Miner, and capturing the Flag ends the game immediately. Placement is its own sub-phase exactly like Battleship's, with one addition: each player can only place within their own three rows, so the click-to-place grid also has to render out-of-zone cells as inert rather than merely un-clickable. Once battle starts, move selection reuses Checkers' existing two-click JS helper (`checkers.js`) completely unmodified, since selecting a square then a destination and filling four hidden form fields has no game-specific logic in it at all.

Nine Men's Morris (`games/logic/nine_mens_morris.py`) breaks two different assumptions every prior game relied on. First, its board isn't a grid: 24 points arranged as three nested squares, connected by corner-to-corner edges along each square's sides plus four radial "spoke" edges linking the squares' midpoints - represented as an `ADJACENCY` dict and a 16-entry `MILLS` list, both *generated* from `ring*8 + position` math rather than hand-typed, specifically to avoid the kind of transcription bug 24 magic-number literals would invite. Rendering follows the same approach: every edge on this board is strictly horizontal or vertical (never diagonal), so the board is plain absolutely-positioned `<div>`s (lines) and `<button>`s (points) at coordinates precomputed from a `POINT_COORDS` unit-grid constant, not the CSS-grid layout every other board game uses, and not SVG either. Second, and more novel: completing a mill (three in a row) doesn't end a turn - it earns an immediate follow-up action (remove one opponent piece) before play passes, so one turn can require two sequential POSTs. This is tracked with a `pending_removal` field holding the id of whoever still owes a removal; `NineMensMorrisMoveView` branches on whether it's set to decide if the incoming POST is a move or a removal, and only advances `match.turn` once no removal is owed - the same kind of turn-assignment exception Rock-Paper-Scissors' simultaneous phase and Battleship/Stratego's placement gate each already needed, just for a "bonus action" rather than a "no turn concept yet." A player reduced to exactly 3 pieces may move to *any* empty point instead of only an adjacent one (the classic "flying" relief rule, kept because it's what keeps the endgame from being a hopeless crawl), and a mill-protected piece can't be removed unless the whole opponent fleet is protected, in which case the restriction lifts - both computed on demand from board state (`removable_points`/`has_legal_move`) rather than tracked incrementally, the same "derive, don't track" discipline as `othello.piece_counts`/`battleship.all_ships_sunk`. Movement-phase piece selection reuses the two-click select-then-destination idea from `checkers.js`, just as a small near-twin (`nine_mens_morris.js`) keyed on a single `data-point` index instead of `data-row`/`data-col`, since the underlying interaction has no shared logic worth abstracting out of a 30-line file.

Backgammon (`games/logic/backgammon.py`) is the app's first game with any randomness in a multiplayer turn â€” every other game is fully deterministic, with dice-adjacent randomness confined to single-player games (Mastermind's secret code, Wordle's target word). It's also the first turn that can require up to four sequential sub-moves rather than Nine Men's Morris' single optional bonus action: `state["dice"]` holds a queue of unconsumed die values (two for a normal roll, four for doubles), `apply_move` consumes exactly one per call, and the view only advances `match.turn` once the queue is empty *or* no remaining die value has any legal use (`any_legal_move`) - a proper action queue rather than a single yes/no bonus flag. Dice are rolled server-side and attached to `match.state` as soon as a turn begins (in the same view call that ends the previous player's turn), so a player never has to separately "roll" - `backgammon.start_turn(state, mover)` rolls and, in the rare case the whole roll has no legal move anywhere, immediately forfeits and rolls for the opponent instead (capped at a few iterations, since a real position essentially never blocks every die value for both sides at once). Backgammon is also the first board to allow stacking - more than one checker of the same color on a point - so each point stores `{"owner", "count"}` instead of the single-piece-per-cell shape every other board game uses, and hitting a lone opponent checker (a "blot") sends it to a `bar` count that must be fully re-entered before that player can make any other move, mirroring Battleship/Stratego's placement-gate pattern but evaluated per-sub-move instead of per-turn. The doubling cube and gammon/backgammon scoring bonuses are deliberately cut, the same kind of simplification as Checkers dropping forced captures â€” win condition is just "bear off all 15 checkers first." The click interaction is a variant of the established two-click pattern: click a checker (or the bar), then click one of the currently-available dice as the distance to move, rather than clicking a destination directly - sidesteps needing a separate UI target for bearing off, since "apply this die from this checker" already produces a bear-off automatically once the destination computes to fall off the board.

`games/stats.py` centralizes the win/loss/high-score queries so both `ProfileView` (which cross-imports from `games`, mirroring the existing precedent of importing from `posts` for karma) and the public `/games/leaderboard/` page read from the same source rather than duplicating aggregation logic. `is_users_turn(match, user)` is the one non-obvious piece of this: it's not a plain `match.turn_id == user.id` check for every game, since Rock-Paper-Scissors has no `match.turn` at all (both players choose simultaneously â€” Â§4) and "your turn" there instead means "you haven't locked in a choice yet." `GamesHubView`'s own your-turn/waiting split had exactly this bug before `is_users_turn` was centralized (every active RPS match landed in "waiting" regardless of whether the viewer had actually picked yet); it and the your-turn badge (Â§5.7) both now call the same helper so they can't disagree with each other.

### 5.5 Editing posts and comments

Both `Post` and `Comment` carry an `edited` boolean (Â§4), shown as "Â· edited" next to the timestamp. This is a plain flag set explicitly in the edit views rather than something inferred by comparing `created_at`/`updated_at` â€” `auto_now_add` and `auto_now` are each stamped independently at save time, so on a brand-new row they can differ by a sub-millisecond amount, which would have produced false "edited" markers on posts nobody ever touched. The flag only flips when `form.has_changed()` is true, so resubmitting a form with unchanged content doesn't mark it edited either.

`PostEditView` already existed as a plain `UserPassesTestMixin` + `UpdateView` before this; it just had no link pointing at it anywhere in the templates. Comments get the same author-only editing from scratch, but as a `CommentEditView(LoginRequiredMixin, View)` mirroring `CommentCreateView`'s POST-only shape (404 for a non-author, rather than pulling in `UserPassesTestMixin`) since there's no GET-rendered page of its own â€” editing happens inline, via the same checkbox+`<label>` CSS-toggle trick `_comment.html` already used for Reply (Â§5.3). Because a comment can now show both a Reply toggle and an Edit toggle at once, both switched to Tailwind's *named* peer modifiers (`peer/reply`, `peer/edit` and their matching `peer-checked/reply:`/`peer-checked/edit:`) instead of a single unnamed `peer` â€” with two same-level checkboxes sharing one unnamed peer class, checking either one would reveal both forms.

### 5.6 Infinite scroll

The feed and a post's comment thread both now render only the first 6 items (`POSTS_PAGE_SIZE`/`COMMENTS_PAGE_SIZE` in `posts/views.py`) and load more as the user nears the bottom, replacing the feed's old Previous/Next pager outright. One small generic script, `posts/static/posts/js/infinite_scroll.js`, drives both: it watches a `[data-infinite-scroll-sentinel]` element via `IntersectionObserver` and, once it's within `rootMargin: "300px"` of the viewport, fetches `data-next-url` with an `X-Requested-With: XMLHttpRequest` header and inserts the JSON response's `html` field just before itself, then updates its own `data-next-url` from the response's `next_url` (or removes itself once that's `null`). Neither view needed a new URL for this â€” `FeedView.render_to_response` and `PostDetailView.get` both just branch on that header and return `JsonResponse({"html": ..., "next_url": ...})` instead of the normal template response, reusing `FeedView`'s existing `paginate_by`/`page_obj` machinery for posts.

Comments paginate differently: over the already-built comment tree in Python, not the underlying queryset, because a top-level comment's replies have to stay attached to it no matter how deep the thread goes, and a query-level `OFFSET`/`LIMIT` over just top-level rows can't also express "and every descendant of these particular rows" without a recursive query. Given this app's comment trees are small, rebuilding the full tree per page is the same proportionate tradeoff `annotate_votes()` already makes by recomputing on every load instead of precomputing (Â§5.2).

One bug this surfaced: the sentinel element originally lived as a *sibling* of `#post-list`/`#comment-list` rather than inside it, so items inserted just before it landed outside those containers â€” outside the reach of `space-y-4`'s margin and `divide-y`'s border, both of which only apply to direct children of the element they're declared on. Fixed by nesting the sentinel as the last child of each container instead.

### 5.7 Keeping multiplayer matches fresh

Multiplayer is still turn-based/asynchronous, not real-time (Â§5.4) â€” these two additions make "it became your turn" visible without a real push channel, not full real-time gameplay. `games/static/games/js/match_poll.js` runs on every match page: it polls the generic `/games/match/<uuid>/status/` endpoint (`MatchStatusView`, one view for all 5 multiplayer games, since the client only ever needs "has anything changed since I loaded this page") every 3 seconds, and reloads the page outright if the returned `updated_at` differs from what was rendered at load time â€” the simplest correct way to show an opponent's move without needing to know how to patch the board's DOM for 5 different games.

The header's Games link also carries a live count badge (`games/context_processors.py`'s `your_turn_count`, added to every page's context so it works from anywhere in the site, not just the games hub) of how many active matches are waiting on the viewer's move. `games/static/games/js/your_turn_badge.js` polls `/games/your-turn-count/` every 10 seconds and updates the badge's text/visibility in place â€” deliberately *not* a page reload like `match_poll.js`, since a global badge reloading the whole site every 10 seconds on every page would be far more disruptive than a single match page doing it only while that one match is still active.

### 5.8 Direct messages

Private one-to-one messaging, added under the `posts` app rather than a new app (posts already owns "content and reactions to it," and a DM is close enough in shape - author, body, timestamp - to fit that umbrella without a fourth top-level app). A "Message" button on another user's profile (`accounts/templates/accounts/profile.html`, alongside Follow/Challenge, hidden on your own profile) POSTs to `StartConversationView`, which resolves the conversation via `get_or_create_conversation()` and redirects straight into it - clicking Message a second time reopens the same thread rather than starting a new one, thanks to `Conversation`'s canonicalized `(user1, user2)` ordering (Â§4).

Privacy is enforced the same way `PostEditView`/`CheckersMatchView` already enforce authorship: `ConversationDetailView` is a `UserPassesTestMixin` `DetailView` whose `test_func` checks `request.user.id in (conversation.user1_id, conversation.user2_id)`, returning 403 for anyone else (checked directly by test, not just implied by "well the view filters it out"). `MessageSendView` does the equivalent check as a plain 404, mirroring `CommentCreateView`'s POST-only, no-GET-page style rather than pulling in the mixin, consistent with how `CommentEditView` made the same call (Â§5.5).

The header's "Messages" link carries the same live-badge treatment as the your-turn badge (Â§5.7): `posts/context_processors.py`'s `unread_messages` makes the count available on every page, and `unread_message_badge.js` polls `/messages/unread-count/` every 10 seconds and updates it in place. One deliberate difference from the your-turn badge: that one disappears entirely at zero, but the Messages link itself always stays visible regardless of count - unlike the games hub (reachable via the always-present "Games" link either way), there'd be no way to open your inbox at all once it hit zero unread if the badge were the only entry point. `ConversationDetailView.get()` marks the other participant's unread messages read *before* building the response context, not after, so this same page load's own badge count already reflects the read state it just caused, rather than showing a stale count until the next page.

Message bodies use the same Markdown pipeline as posts/comments (`{{ message.body|markdown }}`, Â§5.1), not a separate one - including syntax-highlighted code fences. The one wrinkle a chat bubble adds that an article body doesn't: `conversation_detail.html` colors sent/received bubbles with `bg-accent`/`bg-accent/25` backgrounds and a matching text color, but the `prose` typography plugin sets `color: var(--tw-prose-body)` directly on the same element (not just on headings/links/inline code) regardless of the bubble's color - left alone, this collapses every bubble's text to Typography's own fixed light-mode-oriented color rather than the bubble's intended `text-fg`/`text-accent-fg`. Fixed with a `.message-body` rule (`input.css`) forcing `color: inherit` on the element itself and its descendants - *except* a highlighted code block and its contents (`:not(pre.highlight, pre.highlight *)`), which keep Pygments' own multi-color palette rather than collapsing to the bubble's single flat color, since that code sits in its own background chip rather than directly on the bubble. This rule needs `!important`: `.prose`'s own color rule lives in Tailwind's `utilities` cascade layer, which always outranks a same-property declaration in this project's `@layer base` block regardless of selector specificity - `!important` is the one thing that reverses that layer-priority ordering per the CSS Cascade Layers spec, letting a `base`-layer override actually win. Originally only targeted `.message-body`'s *descendants*, which happened to work for the sent-message color (accent-fg, matching what those descendants would've inherited anyway) but silently left received messages unreadably dark-on-dark in dark mode, since `.prose`'s color rule set the wrong color on `.message-body` itself and every descendant just inherited that same wrong color - caught from a live bug report, not by original testing, since the two bugs (the missing `.message-body`-itself target and the layer-priority miss) canceled each other out for sent messages but not received ones.

### 5.9 Simulations

`simulations` is a separate app from `games` on purpose: everything in `games` is user-generated or session-tracked (a `Match`/`SinglePlayerResult` row, or session state), while a simulation is curated example content nobody plays or scores â€” there's no CRUD surface, no user association, and nothing to migrate. `simulations/registry.py`'s `SIMULATIONS` is a plain Python list of dicts (slug, title, description, and a static JS file path) rather than a database model; `SimulationHubView` lists it, `SimulationDetailView` looks up a slug and 404s on a miss. `makemigrations --check --dry-run` stays clean for this app precisely because there's no model to migrate.

Each sketch is a self-contained p5.js file in `simulations/static/simulations/js/`, using p5's "global mode" (`window.setup`/`window.draw`) wrapped in an IIFE so helper state (position, velocity, trail history) doesn't leak into the global scope while still exposing the two globals p5's global-mode detection looks for. p5.js itself is loaded from a CDN (`cdnjs.cloudflare.com`, pinned to a specific version) rather than vendored as a static file â€” the one deliberate exception, alongside the CDN-loaded fonts already used elsewhere, to keeping every other script first-party; vendoring an ~800KB creative-coding library felt like a worse fit for the project's minimal-build-artifact ethos than a pinned CDN `<script>` tag. `detail.html` passes the simulation's `js_file` to Django's `{% static %}` tag as a context variable (unquoted, so it resolves as a variable rather than a literal string) after the CDN tag, so the sketch's `setup`/`draw` are defined only once p5 itself already exists on `window`.

The launch sketches are intentionally simple physics/algorithm toys, not general-purpose demos: Bouncing Ball uses explicit Euler integration (gravity accumulates into velocity, velocity into position) with a velocity-damping bounce off the floor and a full-elastic bounce off the side walls â€” accurate enough for a visual toy, not written for numerical precision. Orbital Gravity is a two-body Newtonian simulation (a fixed sun, one orbiting planet) using the same Euler integration rather than a symplectic integrator like Verlet â€” the standard simplification tradeoff for a short-lived visual demo, where Euler's slow energy drift isn't visible over the timescale anyone would actually watch it. Its initial velocity was chosen deliberately between zero and the circular-orbit speed for its starting distance, so the demo traces a stable, bounded ellipse rather than spiraling into the sun or escaping off-screen.

Boids (`boids.js`) is the first sketch with no physical-body analog and no p5.js state beyond plain objects in an array: each of 80 agents steers by the three classic Craig Reynolds rules â€” separation (steer away from anything within a tight radius), alignment (match the average heading of nearby flockmates), and cohesion (steer toward their average position) â€” computed with an O(nÂ˛) all-pairs scan each frame, which is proportionate at 80 agents and avoids the added complexity of a spatial-partitioning structure a much larger flock would need. Steering force is capped before being added to velocity, and velocity is separately capped to a max speed â€” the standard two-stage limit that keeps a boid from instantly snapping to a new heading. The canvas wraps toroidally (an agent exiting one edge re-enters the opposite one) rather than bouncing or being clamped, since a flock naturally reads better as an unbounded plane than a walled box.

Conway's Game of Life (`game_of_life.js`) drops physics entirely for the first purely discrete-state sketch: a 60Ă—40 grid of cells, randomly seeded, advanced one generation every 10 frames (a plain frame counter, not a separate timer) rather than every frame, since redrawing the same generation for a few frames makes the still-readable board update at a pace a viewer can actually follow rather than flickering. Neighbor counting wraps toroidally, same as Boids, so gliders and other patterns can drift off one edge and reappear at the opposite one instead of dying prematurely against a hard boundary. The whole board is recomputed into a fresh 2D array each generation rather than mutated in place - mutating while reading would let an already-updated neighbor bleed into the same generation's count, corrupting the rules.

Double Pendulum (`double_pendulum.js`) is the app's second Euler-integrated physics sketch after Orbital Gravity, but a chaotic one: two rigid arms swinging under gravity, using the standard closed-form double-pendulum angular acceleration equations rather than a general rigid-body/constraint solver, since a two-link system is small enough that the textbook formulas are simpler than any generalized machinery would be. Unlike Orbital Gravity, the physics runs in unit-length/unit-mass quantities (`L1=L2=1`, not pixels) with a separate `PIXELS_PER_UNIT` scale applied only when drawing - mixing pixel-scale lengths directly into the acceleration formulas would need a wildly different gravitational constant to stay numerically sane. Four physics substeps run per rendered frame at a small time step, the same "keep the integrator stable without adopting a heavier scheme" tradeoff as everywhere else in this app - a double pendulum is chaotic by nature (two runs from barely different starting angles diverge completely), so unlike Orbital Gravity's bounded ellipse, Euler drift here just looks like more chaos rather than a visibly wrong trajectory.

Flow Field (`flow_field.js`) is the app's first use of p5's Perlin `noise()` function: 300 particles each sample a noise value at their own position (plus a slowly-incrementing third "time" coordinate, so the whole field drifts rather than staying frozen) and follow the resulting angle as their heading, wrapping to a random new position whenever they exit the canvas rather than bouncing. The trail effect comes from never calling `background()` after `setup()` - instead, every frame draws a nearly-transparent dark rectangle over the whole canvas before drawing new particle strokes, so old strokes fade gradually rather than being wiped instantly, a standard p5 idiom for trail effects that needed no new architecture beyond what `createCanvas` already provides.

Mandelbrot Zoom (`mandelbrot_zoom.js`) is the app's first sketch driven by direct pixel-buffer math (`loadPixels`/`pixels`/`updatePixels`) rather than p5's shape-drawing API, since a fractal is naturally a per-pixel computation, not a collection of drawable shapes. Every frame recomputes the full escape-time iteration count for all 350Ă—350 pixels around a fixed, visually interesting point, then multiplies the zoom level by a small constant factor before the next frame - a continuous zoom rather than a one-shot render, so the "simulation" still has visible motion like every other sketch in the app, at whatever frame rate the browser can actually sustain doing ~7 million escape-time iterations a frame. Zoom resets back to 1 once it passes a large threshold, since floating-point precision in the pixel-to-complex-plane mapping eventually degrades at extreme zoom - a deliberately abrupt reset rather than a smooth one, the same proportionate-simplification posture as every other sketch's cut corner.

Langton Ant (`langtons_ant.js`) is Game of Life's mirror image: instead of one rule applied to an entire grid simultaneously, a single agent walks a grid one cell at a time, flipping the cell it's standing on and turning based on that cell's *prior* color (clockwise on white, counter-clockwise on black) - deceptively simple rules that produce a few hundred steps of visual noise before settling into an endlessly-repeating diagonal "highway," the classic emergent-complexity result this sketch exists to show. Twenty steps run per rendered frame rather than one, purely so the highway shows up in a watchable amount of time instead of requiring several minutes of real-time stepping; the whole grid is still redrawn from scratch every frame; there was no reason to track a smaller "dirty rectangle" of just the changed cell at this grid size.

Cloth Simulation (`cloth.js`) is the app's first sketch using Verlet integration with iterative constraint relaxation instead of the explicit-Euler-plus-forces approach every other physics sketch uses (Bouncing Ball, Orbital Gravity, Double Pendulum) - a deliberate architectural choice, not just a different formula, because a cloth's defining behavior (many points held at fixed distances from their neighbors) is what Verlet integration and position-based constraints are specifically good at expressing directly, where a force-based spring system would need carefully-tuned stiffness/damping constants to avoid oscillating apart. Velocity is never stored explicitly - each point only remembers its previous position, and velocity falls out implicitly as `current - previous` - so a constraint can move a point without also having to reconcile a separate velocity value, the standard Verlet trick that keeps constraint-solving simple. A `CONSTRAINT_ITERATIONS` relaxation pass runs the same distance-constraint correction over every edge multiple times per frame rather than once, since a single pass only partially satisfies a grid of interlocking constraints - each point is pulled toward satisfying one edge at the expense of slightly violating its neighbors', and several passes are what let the whole grid converge toward a consistent, non-stretchy shape. A small sinusoidal horizontal force (`wind`) is added purely so the settled cloth keeps visibly moving rather than hanging still once gravity and the constraints reach equilibrium - the same "must show visible motion" requirement Mandelbrot Zoom's continuous zoom and Boids' otherwise-stable flock both satisfy differently.

Reaction-Diffusion (`reaction_diffusion.js`) is the app's second sketch built on `loadPixels`/`updatePixels` after Mandelbrot Zoom, but a continuous PDE simulation rather than a one-shot-per-frame formula: two virtual chemical concentrations diffuse across a toroidal grid and react according to the classic Gray-Scott equations, producing the same kind of organic, coral-like patterning Turing's reaction-diffusion model is famous for. The grid is deliberately small (150Ă—150) and simulated at native resolution, then scaled up for display via a CSS `width`/`height` override on the canvas element rather than a larger internal grid - keeping the actual simulation cheap (the discrete Laplacian this needs touches all 8 neighbors of every cell, multiple times per frame) while still rendering at a comfortable on-screen size, the same kind of resolution/performance trade Mandelbrot Zoom already makes by capping its canvas at 350Ă—350. Twelve simulation steps run per rendered frame - a diffusion process is visually slow at one step per frame, so batching steps is what makes the pattern's growth actually watchable rather than imperceptibly gradual.

Fireworks (`fireworks.js`) is the app's first pure particle-lifecycle sketch: unlike every prior sketch's fixed-size cast of persistent bodies (one ball, one flock, one grid), particles here are created continuously (a rocket launches on a timer) and destroyed continuously (an exploded rocket's fragments each carry their own fading `life` value and are spliced out once it hits zero), so the two arrays holding rockets and particles are constantly changing size rather than being allocated once in `setup()`. It's also the app's only sketch in HSB color mode (`colorMode(HSB, ...)`) rather than the RGB values every other sketch uses, since picking a random, vivid, evenly-distributed hue per firework is a one-line `random(360)` in HSB where the RGB equivalent would need real color-space math. The translucent dark-navy `background()` redrawn every frame (rather than a hard clear) is the same fading-trail idiom Flow Field established, reused here to give both rocket ascents and explosion bursts a trailing streak instead of a bare dot.

All eleven sketches respond to mouse and, where relevant, keyboard input via p5's global-mode event hooks (`window.mousePressed`/`mouseDragged`/`mouseReleased`/`keyPressed`), the same naming convention as `setup`/`draw` so no new wiring pattern was needed beyond defining another well-known global. Every handler starts by checking `mouseX`/`mouseY` against `0`â€“`width`/`height` before acting, since these are attached to `window` and fire for clicks anywhere on the page, not just inside the sketch's own canvas. The physics sketches (Bouncing Ball, Orbital Gravity, Double Pendulum) all pause their own integration step while a body is being dragged and read the drag's release velocity from `mouseX - pmouseX`/`mouseY - pmouseY`, so letting go "throws" the body rather than just relocating it - one interaction shape reused three times rather than three different ones. The grid-based sketches (Game of Life, Langton Ant) added a click-to-toggle-a-cell handler plus a spacebar pause, deliberately choosing a single click rather than click-and-drag painting, since painting would need to suppress the toggle-flicker of re-triggering on the same cell every frame the mouse passes back over it. Boids and Flow Field both turn the mouse into a field effect on existing particles (an attraction term added into `computeSteering` while pressed, a repulsion term added onto each particle's velocity within a radius) rather than a discrete action, reusing each sketch's existing per-particle update loop instead of adding a parallel code path. Cloth Simulation's interaction needed the most new logic: a `nearestPointIndex` helper to hit-test the grid of points, and a `draggedPoint` index that both `updatePoints` and `satisfyConstraints` treat exactly like a `pinned` point for the duration of the drag - reusing the existing pinning mechanism instead of inventing a second one - with clicking an actual pinned anchor toggling its `pinned` flag off entirely (cutting it loose) rather than dragging it. Reaction-Diffusion's `mousePressed`/`mouseDragged` seed new chemical `B` at the cursor; because that canvas is drawn at a smaller internal resolution and scaled up via CSS (Â§5.9 above), the natural worry was that `mouseX`/`mouseY` might arrive in the larger *displayed* CSS pixel space rather than the sketch's internal 150Ă—150 grid space - verified empirically (a click 60 CSS-pixels into the canvas reported `mouseX` of exactly 20, matching the 150Ă—450 scale ratio) that p5 already corrects for a CSS-scaled canvas's mouse coordinates on its own, so no manual scaling was needed. Mandelbrot Zoom's click recenters `CENTER_X`/`CENTER_Y` on the clicked point and resets `zoom` to `1`, turning the sketch from a fixed automatic zoom into a click-to-explore fractal browser with no change to the zoom/render logic itself. Fireworks' click launches a rocket immediately from the cursor's x position instead of waiting for the next automatic launch, reusing `makeRocket()` with an optional starting-x argument rather than a separate launch function.

### 5.10 Per-user timezones

Every datetime in the app was rendered in `settings.TIME_ZONE` (`"UTC"`) regardless of who was looking at it, since Django only converts a timezone-aware datetime to a *different* zone when one has been explicitly activated for the current thread (`django.utils.timezone.activate()`) - by default, nothing ever calls that, so every request just falls back to the server's own zone. Fixing this needed exactly two additive pieces, no template changes anywhere: a `Profile.timezone` field (Â§4) storing an IANA zone name, and `accounts/middleware.py`'s `ActivateUserTimezoneMiddleware`, which reads `request.user.profile.timezone` and calls `timezone.activate(ZoneInfo(profile.timezone))` at the start of every request (falling back to `timezone.deactivate()` - i.e. the server's `TIME_ZONE` - for anonymous visitors or the rare user with no profile row). Registered directly after `AuthenticationMiddleware` in `MIDDLEWARE`, since it needs `request.user` to already be resolved.

Every template across the app already renders datetimes as a bare `{{ post.created_at }}`/`{{ message.created_at }}` with no explicit format filter (checked directly, not assumed) - and Django's template variable resolution automatically runs an aware datetime through `timezone.template_localtime()` before formatting it, converting to whichever zone is currently active. That's the whole reason this feature needed no template edits at all: activating the right zone once per request is sufficient for every existing `created_at`/`updated_at` display (feed, post detail, comments, direct messages, profile post/comment lists) to start rendering correctly, all at once.

`Profile.timezone` is a plain `CharField` rather than a `TimeZoneField`-style custom field or a third-party dependency, with `choices` built once at import time from `zoneinfo.available_timezones()` (`accounts/models.py`'s `TIMEZONE_CHOICES`, sorted) - Python's standard library has shipped IANA timezone data since 3.9, so this needed no new dependency, and Django's `ModelForm` machinery turns a `CharField` with `choices` into an ordinary `<select>` in `ProfileForm` (`fields = ["bio", "avatar", "timezone"]`) with zero new form or template code, appearing on `/settings/profile/` for free via the existing `{{ form.as_p }}`. A ~600-option flat `<select>` is unfiltered/unsearched, which is a real UX rough edge at this list length, but proportionate to this app's scale rather than justifying a JS-enhanced combobox (the "no JS framework" stance, Â§1).

The one fix required in existing tests: `ProfileEditTests`' POST payloads never included a `timezone` value, and adding a `choices`-restricted required field to `ProfileForm` meant every one of those submissions started failing validation silently underneath the specific assertion each test cared about. Fixed by adding `"timezone": "UTC"` to each payload, mirroring the same completeness the tests already gave `username` (itself unrelated to what most of those tests check) rather than special-casing the new field to be optional.

### 5.11 User tagging (`@mentions`)

Originally built on a `feature/user-tagging` branch to try out, since merged to `main`. Two independent pieces: every displayed username now renders with a leading `@`, and post/comment textareas offer a live autocomplete when typing one.

The `@` prefix is a single template filter, `accounts/templatetags/user_extras.py`'s `at_username`, applied at every *display* site across the app (feed, post detail, comments, direct messages, profile heading/title, the header nav link, game match pages, the leaderboard's `values(...).annotate(...)` rows) - not at every `.username` reference, since plenty of those are non-displayed identifiers (`{% url 'profile' post.author.username %}` href args, the avatar's CSS-gradient hash, the avatar's single-letter initial) that would look wrong or simply don't need it. No model or middleware changes were needed - this is a purely template-layer, filter-based change, the same shape as `avatar_gradient` in the same app.

The autocomplete (`posts/static/posts/js/mention_autocomplete.js`) is a single vanilla-JS file using event delegation on `document` for `input`/`keydown`/`click` - not per-textarea listeners - so it automatically covers every current and future `textarea[name="body"]` on the page without needing to know in advance how many exist: the main post/comment forms (each rendered once) and every nested comment's Reply/Edit textareas in `_comment.html` (rendered once per comment, however deep the thread goes). On each keystroke it checks the text immediately before the cursor for a still-being-typed mention (`/(?:^|\s)@(\w*)$/` - an `@` preceded by whitespace or start-of-text, followed by word characters up to the caret), debounces 200ms, then queries `accounts.views.UserSearchView` (`GET /users/search/?q=<partial>`, `username__istartswith`, capped at 8 results, login-required like every other write-adjacent endpoint in the app) and renders a small absolutely-positioned dropdown under the textarea. Clicking a suggestion uses a `mousedown` handler with `preventDefault()` (not `click`) specifically so the textarea never loses focus before the replacement happens, then splices `@username ` into the textarea at the mention's start position.

`/users/search/` had to be registered *before* `/users/<str:username>/` in `accounts/urls.py` - Django tries URL patterns in list order, and the existing profile-lookup pattern would otherwise have swallowed `/users/search/` as a request for a user literally named "search". Caught by a dedicated test (`test_anonymous_user_cannot_search` asserting a redirect to login, which only happens if the request actually reached `UserSearchView` and not `ProfileView`) rather than assumed from reading the code.

`templates/_footer.html`, included once from `base.html` (the same modular-partial convention as `_header.html`), shows a copyright line and, for logged-in users, a "Send us a message" action. The year is rendered with `{% now "Y" %}` rather than hardcoded, so it never needs a yearly edit.

"Send us a message" reuses the existing DM machinery wholesale (`StartConversationView`, the same `POST`-only view a profile page's own "Message" button already hits, Â§5.8) rather than adding a new contact-form path - it's just a `<form>` whose `action` targets whichever user is the site's admin. ### 5.13 Notifications (mentions, replies, upvotes)

A single `Notification` model (`posts/models.py`) - `kind` (`mention`/`reply`/`upvote`), `recipient`, `actor`, `post` (always set), and `comment` (only set when the notification is about a comment rather than the post itself, since either way `post` is the natural landing page). Launched mention-only with no `kind` field at all; `kind` was added in a follow-up migration with `default="mention"`, which backfills every existing row correctly for free since they were all mentions so far. Still not a generic polymorphic notification type for arbitrary future event kinds (follows, etc.) - each addition so far has fit the same `recipient`/`actor`/`post`/`comment` shape without needing a schema change, so there's been no actual need for more generality yet.

`posts/mentions.py`'s `extract_mentioned_users(body, exclude=None)` finds `@username` tokens (`MENTION_RE = r"@([\w.+-]+)"`, the same character set the username validator allows minus `@` itself, since `@` can't appear inside a real username and so unambiguously starts a mention token) and resolves them to real users, case-insensitively - a mention typed in the wrong case should still resolve, the same posture `UserSearchView`'s `istartswith` already takes. The case-insensitive match is a single `Q`-OR'd query across every found token, not a per-token loop with a "did the exact-case query come up empty" shortcut - an early draft took that shortcut and had a real bug: a mixed-case body (a correctly-cased mention alongside a wrong-case one) would skip the fallback entirely, since it only ran when *no* usernames matched exactly, silently dropping the wrong-case mention rather than catching it. Caught before shipping by re-reading the draft rather than by a failing test.

Notifications are created only when a post or comment is first **created** (`PostCreateView.form_valid`, `CommentCreateView.post`), never on edit - a deliberate simplification. Diffing old vs. new mentions to avoid re-notifying for ones that were already there is real complexity for a case (adding a new mention when editing) that's rare enough not to justify it up front.

The badge/list machinery is a straight copy of the unread-message pattern (Â§5.8): `unread_notification_count(user)`, a `posts.context_processors.unread_notifications` context processor, `NotificationListView` (marks everything read *before* rendering, not after, so this same response's own badge count already reflects it - the same ordering `ConversationDetailView` uses and for the same reason), a JSON `/notifications/unread-count/` endpoint, and `notification_badge.js` polling it every 10s - reusing three or four already-established shapes rather than inventing a new one.

**Replies** notify whoever's being responded to - a reply to a comment notifies that comment's author, a top-level comment (no `parent`) notifies the post's author - one rule instead of two separate code paths. The stored `comment` on a reply notification is the *new* reply itself (so the link jumps straight to it), which meant the template's post-vs-comment wording couldn't just check "is `comment` set" the way mention/upvote notifications do (it's *always* set for a reply) - it has to check `notification.comment.parent_id` instead, to distinguish "replied to your comment" from "replied to your post."

**Upvotes** required teaching `toggle_vote` (`posts/views.py`) to report what it did - `"created"`, `"flipped"`, or `"removed"` - since only two of those three outcomes should notify (a brand-new upvote, or an existing downvote flipping to one; removing an upvote, or downvoting, never notifies). `PostVoteView`/`CommentVoteView` check that return value before calling `create_notification`. Self-upvotes stay silent for two overlapping reasons: the automatic self-upvote-on-your-own-content signal (`posts/signals.py`) creates its vote directly via `get_or_create`, bypassing `toggle_vote` (and this new code) entirely; and `create_notification` itself refuses to notify when `recipient == actor`, as a second, explicit guard in case a user ever re-upvotes their own content through the actual button. Deliberately not de-duplicated: upvote â†’ remove â†’ upvote again produces two separate notification rows rather than one bumped-timestamp row - the simpler behavior, with the tradeoff written down here rather than silently accepted.

A real bug surfaced while wiring up the reply/upvote wording in `notification_list.html`: a multi-line explanatory note was written as `{# ... #}`, Django's *single-line* comment syntax, which doesn't strip content that spans multiple lines - it rendered as literal visible text on the page. `{% comment %}...{% endcomment %}` is the correct tag for a multi-line note. Caught by loading the actual page in a browser after building, not by any test (a test asserting the expected wording still passed - the leaked comment text just appeared *before* the correct wording rather than instead of it) - a regression test was added afterward asserting the comment text specifically does *not* appear, which a wording-only assertion can't catch.

A `dismissed` boolean (not a new model, not a soft-delete via `deleted_at`) lets a user hide a notification from the list without ever actually removing the row - `NotificationDismissView` (POST-only, scoped to `recipient=request.user` via `get_object_or_404` so dismissing someone else's notification 404s rather than leaking its existence) just flips it and redirects back to the list, which then filters `dismissed=False`. "Only after they've been read" needed no explicit gate: `NotificationListView.get()` already marks every notification read *before* rendering the page (Â§5.13 above), and the Dismiss button only exists on that same rendered page - so by construction, nothing reachable by the dismiss button was ever unread. Templates required restructuring, not just a new button: the row had been one big `<a>` wrapping the whole card, and a `<form><button>` can't validly nest inside an `<a>` - split into two siblings (the link, and the dismiss form) inside a shared flex container instead.

`NotificationDismissAllView` is a one-line bulk version of the same idea - a single `.update(dismissed=True)` scoped to `recipient=request.user`, no per-row loop - and the "Dismiss all" button next to the page heading is wrapped in `{% if notifications %}` so it only appears when there's anything to dismiss.

### 5.14 Post and comment deletion

Soft delete, not a real row deletion - a `deleted` boolean on both `Post` and `Comment`, and the actual `body` text is cleared to `""` at the same time (not just flagged), so deleted content doesn't sit around indefinitely just because a UI flag is hiding it. The reason this has to be soft: `Comment.parent` is self-referential `on_delete=models.CASCADE`, so actually deleting a comment row would cascade-delete every reply underneath it too - someone else's replies vanishing just because the original commenter deleted theirs. `Comment.post` is `CASCADE` the same way, so a post's comments would suffer the identical problem. Soft delete sidesteps both: the row (and its children) survive, `PostDetailView`'s comment tree keeps rendering exactly as before with no change to `build_comment_tree` at all, and only the template layer needs to know about `deleted`.

Three template spots swap the real body for a literal "[deleted]": `post_detail.html`'s full body, `_post_card.html`'s `post.first_line` (which would otherwise just silently render as an empty string once `body` is cleared - easy to miss since nothing would *look* broken, just blank), and `_comment.html`'s body - with the `{% for child in comment.children %}` loop directly beneath left completely untouched, since replies rendering normally underneath a deleted parent is the entire point of choosing soft delete in the first place.

`PostDeleteView` and `CommentDeleteView` mirror the two different permission-check idioms the existing edit views already established, rather than inventing a third: `PostDeleteView` uses `UserPassesTestMixin` (403 for a non-author) like `PostEditView`; `CommentDeleteView` does the same manual `if comment.author_id != request.user.id: raise Http404` `CommentEditView` already does (404, not 403 - an existing asymmetry between the two views, kept rather than "fixed" as an unrelated change). Both views also add `and not deleted` (Post) / `or comment.deleted` (Comment) to their edit-view's permission check, so an already-deleted post/comment can't be edited back to life.

No confirmation dialog on Delete - a plain one-click `<form>`/button, matching every other destructive-ish action already in the app (Unfollow, flipping a vote, starting a conversation are all single-click with no confirm step). A deliberate, explicitly-flagged judgment call rather than an oversight.

### 5.15 Muting and blocking

Two separate models (`accounts/models.py`, alongside `Follow`) rather than one shared `Relationship`-with-a-`kind` model like `Notification` uses - mention/reply/upvote all have *identical* effects, so a shared `kind` there just picks between otherwise-identical code paths; `Mute` and `Block` have genuinely different effects (Block also cuts off DMs, Mute doesn't), so a shared model would still need `if kind == "block"` branching at the one place they diverge, without actually saving any duplication. Both mirror `Follow`'s exact shape (UUID pk, `UniqueConstraint`, a `CheckConstraint` against self-muting/self-blocking).

`is_muted_or_blocked(viewer, other)` is the one shared helper for everywhere Mute and Block have the *same* effect - checked in `FeedView.get_base_queryset` (excludes muted/blocked authors' posts from the viewer's feed only, not anyone else's) and in `create_notification`/`notify_mentioned_users` (`posts/views.py`) before creating a mention/reply/upvote notification. `is_blocked_either_way(user_a, user_b)` is separate and Block-only, since Mute never restricts messaging - it's checked in `StartConversationView` (a fresh conversation) and `MessageSendView` (continuing one that predates the block), covering both directions with a single `Q(blocker=a, blocked=b) | Q(blocker=b, blocked=a)` query rather than two separate checks.

Deliberately **not** mutual profile/content hiding: blocking someone doesn't stop them from viewing your profile or posts, or from commenting on them - it only stops their content from showing back up in *your* feed/notifications, and stops new DMs between you. A fully mutual wall would mean touching `ProfileView`, comment rendering, and the vote views too, for a first pass that's really about the two things people block for most: unwanted DMs and unwanted feed/notification noise. Written down here as a scope decision, not a gap that was missed.

`profile.html`'s Message and every "Challenge to X" button share one `{% if not is_blocked_either_way %}` wrapper rather than each getting their own copy of the same condition - the block state doesn't vary between them, so one wrapper around the whole group is simpler than repeating it eleven times. Mute/Unmute and Block/Unblock buttons are unconditional (always visible, independent of each other and of the follow state), since you should always be able to toggle either regardless of whether you're also following, muted, or already blocked in the other direction.

### 6a. Deploying to Render + Neon

The app is deployable for free without any code forking: **Render** runs the existing `Dockerfile` directly as a Docker-native web service, and **Neon** provides serverless Postgres â€” both have generous, non-time-limited free tiers, unlike most "free trial" hosts. `render.yaml` is a Blueprint that provisions the web service and generates `SECRET_KEY` automatically; `DATABASE_URL`, `ALLOWED_HOSTS`, and the optional email/OAuth/S3 vars are all `sync: false` (secrets or values Render can't know upfront, filled in via its dashboard instead â€” see the README's step-by-step).

A few settings exist purely to make this work correctly rather than just "work":

- `dj_database_url.parse(..., conn_max_age=0, ssl_require=True)` for `DATABASE_URL` â€” `conn_max_age=0` (no persistent connections) is deliberate, not an oversight: Neon's commonly-copied connection string routes through PgBouncer in transaction-pooling mode, and Django holding a connection open across requests on top of that can surface as "prepared statement already exists" errors or session state leaking across unrelated requests.
- `IS_RENDER` (from Render's auto-injected `RENDER=true` env var) gates `SECURE_SSL_REDIRECT`/`SECURE_PROXY_SSL_HEADER`/HSTS specifically, rather than gating them on `DEBUG=False` the way the generic cookie-security settings are. The distinction matters: "definitely behind Render's TLS-terminating edge" and "`DEBUG=0` for some other reason" aren't the same thing, and forcing an HTTPS redirect with no HTTPS listener actually in front (e.g. `DEBUG=0` used locally) would make the app completely inaccessible.
- Avatar storage defaults to local disk, which doesn't survive a redeploy on Render's free tier (ephemeral disk). Setting `AWS_STORAGE_BUCKET_NAME` (plus the matching key/secret/endpoint vars) switches `STORAGES["default"]` to `django-storages`' S3 backend, which also works against any S3-compatible host (Cloudflare R2, Backblaze B2) via `AWS_S3_ENDPOINT_URL` â€” left unset by default so local dev and single-container deploys keep the simpler disk backend.
- `ALLOWED_HOSTS` can't be known before the first deploy (Render assigns the hostname), so it's `sync: false` and the README documents setting it *after* first deploy â€” until then, Django rejects every request including Render's own health check with a 400, which is expected and self-resolves once it's set.

Both Render's free web service and Neon's free compute spin down/autosuspend after a period of inactivity, so the first request after a quiet period pays a cold-start delay â€” this doesn't affect correctness, just documented as an expectation in the README.

## 10. Risks and open decisions

Comment-tree performance is still the main scaling caveat, addressed in Â§4 with a clear upgrade path (recursive CTE or django-mptt) if it's ever needed â€” infinite scroll (Â§5.6) caps how much of the tree renders per request, but still rebuilds the *entire* tree in Python on every page fetched, so it doesn't reduce the underlying query/build cost for a post with a huge thread, only the render cost. Avatar storage on local disk is the default â€” noted in the README, S3-compatible storage via `django-storages` is the drop-in fix (Â§6a) if this deploys somewhere with ephemeral or multi-instance disk. Static files are served by Django in dev; WhiteNoise handles it in production (Â§6a). Games' turn-based design means there's still no notification beyond visiting the site or having a match page open (Â§5.7's polling only helps once you're already looking at the relevant page) â€” a natural future addition (email digest, or a browser push) if the games see real use; match/badge polling also means every open match page and every page load makes a small recurring background request, fine at this app's scale but worth revisiting (e.g. WebSockets/SSE) if concurrent active matches grew large. Feed ranking (Â§5.2) recomputes over every matching post in Python on each load rather than pre-sorting at the database level, to avoid a join-fan-out bug that a single cross-relation `annotate()` would hit â€” fine at this app's post volume, but would need a cached/precomputed rank (e.g. a periodic job writing a `rank_score` column) if post counts grew much larger. UUID primary keys (Â§4) trade a small amount of index/storage overhead and non-sequential insert locality for unguessable, non-enumerable URLs â€” the right tradeoff at this app's scale, and Postgres's native `uuid` type keeps the overhead modest (16 bytes, not a 36-character string). Direct messages (Â§5.8) are strictly one-to-one by design, not a stepping stone toward group chat - `Conversation`'s two-FK shape and `Message.read`'s single boolean both assume exactly two participants and would need a real rework (a participants M2M, per-recipient read receipts) rather than a small extension if group messaging were ever added. The unread-message badge adds one more small recurring background request per page load on top of the your-turn badge's (Â§5.7) - fine at this app's scale, same caveat as that one about revisiting if it ever needed to scale further.
